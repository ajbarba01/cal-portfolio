Inquiries — design mockup

One shared component set, two consumers. The account view (client owns their messages → can edit) and the restructured admin view (Cal reads + replies, can't edit a client's words). Warm Trail palette, Fraunces headings, Public Sans body — tokens straight from globals.css.

1 Account · list view

Search filters live. Segmented control toggles status. Cards are a uniform grid of fixed-height tiles — subject clamps to one line, body to three with a soft cut-off. Sorted newest-first. The whole tile is a button; the inline actions stop propagation.

4 Resolve · confirm dialog

Resolve is irreversible, so it always routes through a confirm. From a tile → confirm opens directly. From the detail popup → the popup closes first, then this appears. Same dialog in both views.

5 Admin · same components, restructured

Modularity: identical tile grid, search, pager, and detail popup. The differences are passed as props — admin tiles surface the client's name + email, the detail popup swaps Edit for reply actions (Email / Text), and edit mode is never mounted. Cal can resolve but never rewrites a client's words.

6 Mobile · parity, not a fallback

The overflow bug, fixed: today a long inquiry's text flows straight out of its box. Here every tile is the same height with subject clamped to one line and body to three (a soft fade marks the cut-off), and the popup body scrolls inside the dialog instead of pushing the layout. Nothing can ever escape its container.

On mobile the grid collapses to a single column, the toolbar stacks (full-width search, scrollable segmented control), and the detail popup becomes a bottom sheet that fills the width. Key rule: on mobile the tiles carry no action buttons at all — tapping a tile opens the sheet, and every action (Resolve, Edit for account; Email / Text / Resolve for admin) lives there as full-width, thumb-reachable buttons. Desktop keeps the inline tile actions; mobile defers them to the sheet. Same components, responsive props.
